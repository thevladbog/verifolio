# Verification Signals

## Purpose

Verification signals describe specific evidence about a user, recommender, document, file, organization, or signature.

Verifolio must not use a vague `verified = true` model.

This document is the authoritative signal catalog. It is the single source of truth for signal types; `DATA_MODEL.md` references this catalog and must not diverge from it.

## VerificationSignal Model

```text
VerificationSignal
- id
- entity_type
- entity_id
- signal_type
- status
- evidence_json
- provider
- verified_at
- expires_at
- created_at
```

`entity_type` uses the canonical `EntityType` enum defined in `DATA_MODEL.md`.

## Statuses

```text
PENDING
VERIFIED
FAILED
EXPIRED
REVOKED
NOT_APPLICABLE
```

### REVOKED Semantics

A `REVOKED` signal was previously `VERIFIED` but was explicitly invalidated. Revoked signals must never be displayed as confirmed and must not count toward the trust summary.

A signal can be revoked by:

- the recommender, through retraction of their response or withdrawal of consent;
- an admin, through a controlled support action that always emits an audit event;
- the system, automatically, when a related consent is withdrawn (`ConsentRecord.status = WITHDRAWN`).

## Core Signal Types

### EMAIL_CONFIRMED

Meaning: the actor confirmed access to an email address.

Evidence:

- email domain;
- confirmation timestamp;
- token purpose;
- provider.

Public text:

```text
Email address was confirmed.
```

Limitation:

```text
Email confirmation proves access to the mailbox, not identity or authority.
```

Expiration: never expires; displayed as "as of <date>".

### PHONE_CONFIRMED

Meaning: the actor confirmed access to a phone number via a one-time code.

Evidence:

- phone country code;
- masked phone number;
- confirmation timestamp;
- provider.

Creator: system, after successful OTP verification.

Public text:

```text
Phone number was confirmed.
```

Limitation:

```text
Phone confirmation proves access to the number at the time of confirmation, not identity or authority.
```

Expiration: never expires; displayed as "as of <date>".

### CORPORATE_DOMAIN_CONFIRMED

Meaning: recommender email belongs to a corporate/domain email.

Determination method: at minimum, the email domain is checked against a deny-list of free and disposable email providers (e.g. gmail.com, mail.ru, temporary-mail services). A domain that is not on the deny-list is treated as corporate. Stronger checks (MX/organization matching) may be added later.

Organization name source: the organization name shown with this signal is recommender-stated unless a verified `Organization` record exists for the domain.

Evidence:

- email domain;
- domain match;
- organization name if available (with its source: recommender-stated or verified record).

Public text:

```text
Corporate domain was confirmed.
```

Limitation:

```text
Domain confirmation is a point-in-time check: it shows the email belonged to the domain when confirmed. It does not prove current employment, and it does not prove the person is authorized to speak on behalf of the company.
```

Expiration: never expires; displayed as "as of <date>".

### RECIPIENT_CONFIRMED

Meaning: recommender confirmed that the document is about the named recipient.

Public text:

```text
Recommender confirmed the recipient.
```

Expiration: never expires; displayed as "as of <date>".

### RECOMMENDER_RELATIONSHIP_CONFIRMED

Meaning: recommender stated and confirmed their relationship to the recipient.

Examples:

```text
Direct Manager
Peer
Client
Professor
HR Representative
```

The relationship value is self-declared and must be labeled "stated by recommender" wherever it is displayed.

Expiration: never expires; displayed as "as of <date>".

### SCAN_ATTACHED

Meaning: a scan or PDF was attached.

Public text:

```text
Source scan was attached.
```

Limitation:

```text
A scan can increase confidence but does not automatically prove authenticity.
```

Expiration: never expires; displayed as "as of <date>".

### SIGNATURE_ATTACHED

Meaning: a signature file was attached.

Evidence:

- `target_file_id` — the `FileObject` the signature covers;
- signature file id;
- signature format.

Public text:

```text
Digital signature file was attached.
```

Expiration: never expires; displayed as "as of <date>".

### SIGNATURE_VERIFIED

Meaning: the attached signature was successfully verified by a supported provider/process.

Evidence:

- `target_file_id` — the `FileObject` the signature covers (the uploaded scan, never the generated PDF);
- signature format;
- certificate subject;
- certificate issuer;
- verification timestamp;
- file hash (`FileObject.sha256_hash` of the target file).

Limitation:

```text
Signature verification proves file integrity and certificate ownership where applicable, and it covers only the uploaded signed file (the scan), not the PDF generated by Verifolio. It does not independently guarantee that every statement in the document is true.
```

Expiration: never expires; displayed as "as of <date>". The signal may become `EXPIRED` or `REVOKED` if the certificate is later found revoked.

### VERSION_LOCKED

Meaning: document version was locked and cannot be silently modified.

Public text:

```text
Document version is locked.
```

Expiration: never expires; displayed as "as of <date>".

### DOCUMENT_HASH_LOCKED

Meaning: file/document hash was calculated and stored.

Public text:

```text
Document hash was recorded.
```

Expiration: never expires; displayed as "as of <date>".

### IDENTITY_VERIFIED

Meaning: user profile identity was verified through an approved process.

This may be implemented later.

Expiration: never expires; displayed as "as of <date>".

### NAME_MATCH

Meaning: the recipient name in the document was compared with the profile's verified name.

Match levels:

```text
STRONG   — names match, including known transliteration variants
PARTIAL  — names partially match (e.g. missing middle name, married-name change, ambiguous transliteration)
NONE     — names do not match
```

Determination rules:

- transliteration between Cyrillic and Latin scripts is handled (e.g. "Богатырев" ↔ "Bogatirev"/"Bogatyrev");
- known married-name changes may downgrade a mismatch to `PARTIAL` when supporting context exists;
- weak matches (`PARTIAL` or `NONE`) are surfaced to the user first, with a dispute path to provide clarification or evidence;
- only `STRONG` matches are displayed publicly; `PARTIAL` and `NONE` states are never shown on public pages.

Evidence:

- compared name values (hashed or redacted where required);
- match level;
- transliteration scheme used;
- comparison timestamp.

Creator: system, at document review time.

Public text (STRONG only):

```text
Recipient name matches the profile name.
```

Limitation:

```text
Name matching compares written names only. It does not prove identity by itself.
```

Expiration: never expires; displayed as "as of <date>". Re-evaluated if the profile name changes.

### PUBLIC_VERIFICATION_ENABLED

Meaning: document has an active public verification page or share link.

Expiration: reflects the current share link state; becomes `REVOKED`/`EXPIRED` when the link is revoked or expires.

## Trust Summary

Verifolio never shows a numeric trust score, percentage, or any single aggregated number.

Trust is presented as:

- the list of signal badges for the entity;
- a derived "trust summary": counts of confirmed (`VERIFIED`) signals grouped by category.

Derivation rule:

```text
For each category (identity, relationship, document integrity, signature, publication),
count signals with status = VERIFIED.
Display the counts per category.
Never reduce the counts to a single number or percentage.
```

Only `VERIFIED` signals count. `PENDING`, `FAILED`, `EXPIRED`, and `REVOKED` signals are excluded from the summary.

Self-declared fields (e.g. recommender-stated relationship or organization name) must be labeled "stated by recommender" and are never presented as verified.

## Public Display Rules

Each signal shown publicly must include:

- plain-language name;
- verified/failed/expired state;
- date;
- explanation;
- limitation if necessary.

## Adding New Signals

When adding a signal:

- define meaning;
- define evidence;
- define who/what can create it;
- define expiration behavior;
- define public display text;
- define limitations;
- add audit event;
- add tests.

## AI-Agent Rule

Do not add new signal types without updating this document.
